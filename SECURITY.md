# SECURITY.md

---

## Reporting a vulnerability

Security disclosures are handled by the human operator, not by agents.

**Do not open a public GitHub issue for security vulnerabilities.**

Report security issues to: security@karta.build

Include in your report:
- Description of the vulnerability
- Steps to reproduce
- Potential impact
- Any suggested mitigations

You will receive acknowledgement within 48 hours and a full
response within 5 business days.

---

## Security policy for agent-generated code

### What we check automatically

Every PR is scanned by CI for:
- Hardcoded credentials and API keys (using truffleHog)
- Common secret patterns in prompt logs before they are committed
- Known vulnerable dependency versions (using Dependabot)
- OWASP top-10 patterns in generated code (using Semgrep)

### What we cannot guarantee

Karta ships software generated by AI agents. Despite automated
scanning, we cannot guarantee that generated code is free of:
- Subtle logic vulnerabilities
- Inadvertent information disclosure
- Insecure default configurations

Users incorporating Karta's software into production systems should
conduct their own security review.

### Prompt injection

This project is specifically aware of prompt injection as an attack
surface. Malicious GitHub issues designed to manipulate agent behavior
are treated as security incidents and logged in INCIDENTS.md.

If you discover a prompt injection attack that successfully manipulated
agent behavior in a merged commit, please disclose it via the address
above. These disclosures are especially valuable to our research.

## Security release process

When a vulnerability is confirmed:

1. Human operator is notified immediately
2. Karta-0 flags the fix PR with [SECURITY-RELEASE] in DECISIONS.md
3. 24-hour hold for human operator review (see MANDATE.md)
4. If no operator veto, Karta-0 merges and tags a patch release
5. SBOM and Sigstore signature generated for the release artifact
6. Public disclosure after patch is available

We follow responsible disclosure: we will not disclose vulnerability
details publicly until a patch is available, and we will credit
reporters who request it.

---

## Supported versions

Karta supports the latest released version only.
Older versions do not receive security patches.

---

## Supply chain verification

Every Karta release is:
- Signed with Sigstore/cosign using GitHub Actions OIDC identity
- Accompanied by a Software Bill of Materials (SBOM)
- Built reproducibly from the tagged commit

To verify a release:
```bash
cosign verify-blob \
  --certificate-identity \
    "https://github.com/karta-oss/karta/.github/workflows/release.yml@refs/tags/v*" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  --bundle karta-<version>.bundle \
  karta-<version>.tar.gz
```
